# 笔记功能设计

**日期：** 2026-04-12

**范围：**
- 电脑端新增笔记列表、笔记详情、笔记编辑、导入与状态恢复
- 手机端新增笔记列表、笔记详情、笔记编辑、多选删除、语音输入
- 手机端与电脑端支持双端离线编辑与准实时跨端同步

**目标：**
- 支持电脑端与手机端分别记录笔记
- 支持双端离线编辑，联网后自动同步
- 冲突不覆盖，保留两份内容
- 保持当前项目现有技术栈与交互模式，避免引入额外服务端

---

## 1. 背景与约束

当前电脑端项目位于 `c:\code\codex1`，主程序以 `wxPython` 桌面界面和本地状态文件为主，已经存在远程控制与状态推送能力。手机端项目位于 `c:\code\rc`，主程序为 Flutter，已经存在底部标签、远程连接、WebSocket、语音输入与本地配置持久化能力。

本次笔记功能必须满足以下约束：
- 电脑端与手机端都能在本地独立使用
- 两端都支持离线新增、编辑、删除
- 重新联网后自动同步
- 同一条内容发生并发编辑时不覆盖，生成冲突副本
- 尽量复用现有远程控制链路，不单独引入云数据库或新部署服务

---

## 2. 方案对比

### 方案 A：电脑端唯一主库，手机端仅缓存

手机端只做浏览和轻量编辑，所有权威数据都以电脑端为准。

优点：
- 实现最简单
- 冲突处理最少

缺点：
- 不满足双端离线编辑目标
- 手机端体验受电脑端在线状态限制

### 方案 B：双端本地库 + 电脑端内置同步服务

电脑端和手机端各自保留完整本地库，电脑端扩展 notes 同步接口，手机端复用现有远程连接能力进行准实时增量同步。

优点：
- 满足双端离线编辑
- 复用现有电脑端远程服务与手机端 WebSocket 基础设施
- 不需要新增独立服务部署
- 可以先做本地功能，再逐步接入同步

缺点：
- 需要设计同步协议和冲突处理规则
- 电脑端需要承担同步中枢职责

### 方案 C：引入独立云端数据库

两端都直连云端统一数据库，电脑端不再作为同步中枢。

优点：
- 多端同步逻辑更标准
- 后续扩展 Web 或更多设备更容易

缺点：
- 与现有项目架构不匹配
- 需要新增部署、鉴权、网络安全和运维成本
- 对当前目标明显过度设计

### 结论

采用 **方案 B：双端本地库 + 电脑端内置同步服务**。

---

## 3. 总体架构

整体采用“本地优先、增量同步、定期校准”的设计。

架构要点：
- 电脑端和手机端都维护完整的本地笔记数据库
- 所有本地操作先写入本地数据库，再写入同步队列
- 联网状态下立即推送增量操作，实现准实时同步
- 每端保存最近同步游标，用于增量拉取
- 定时执行轻量全量校准，修复事件漏收或游标不一致
- 并发修改同一对象时生成冲突副本，不做自动覆盖

电脑端角色：
- 本地笔记存储
- 笔记界面与键盘交互
- notes 同步 API / WebSocket 事件源
- 冲突判定与冲突副本生成

手机端角色：
- 本地笔记存储
- 笔记页、详情页、编辑页与语音输入
- outbox 推送与增量拉取
- 同步状态提示与冲突结果展示

## 4. 数据模型

建议两端统一使用 SQLite 存储笔记主体数据，JSON / SharedPreferences 仅保存界面恢复状态和同步游标。

### 4.1 Notebook

表示笔记列表中的一个笔记项目。

字段：
- `id`：全局唯一 ID，建议使用 UUID
- `title`：笔记标题
- `created_at`：创建时间，UTC
- `updated_at`：最后修改时间，UTC
- `deleted_at`：软删除时间，未删除为空
- `pinned`：是否置顶
- `sort_order`：排序权重
- `version`：对象版本号，每次本地修改递增
- `device_id`：最近一次修改来源设备 ID
- `last_modified_by`：`desktop` 或 `mobile`
- `is_conflict_copy`：是否冲突副本
- `origin_notebook_id`：若为冲突副本，记录原始笔记 ID

### 4.2 NoteEntry

表示某个笔记详情中的一条内容。

字段：
- `id`
- `notebook_id`
- `content`
- `created_at`
- `updated_at`
- `deleted_at`
- `pinned`
- `sort_order`
- `version`
- `device_id`
- `last_modified_by`
- `is_conflict_copy`
- `origin_entry_id`
- `source`：`manual`、`import_file`、`import_clipboard`、`voice`

### 4.3 SyncOutbox

表示待同步操作，用于断网重试和启动后恢复推送。

字段：
- `op_id`
- `entity_type`：`notebook` 或 `entry`
- `entity_id`
- `action`：`create`、`update`、`delete`、`reorder`、`rename`
- `payload_json`
- `base_version`
- `created_at`
- `retry_count`
- `status`：`pending`、`sending`、`acked`、`failed`

### 4.4 UiState

仅保存界面恢复所需状态，不保存完整笔记主体。

字段：
- `active_root_tab`
- `notes_view`
- `active_notebook_id`
- `active_entry_id`
- `entry_editor_draft`
- `entry_editor_dirty`
- `entry_editor_cursor`
- `entry_editor_scroll`
- `last_sync_cursor`

## 5. 同步模型

### 5.1 同步原则

- 本地编辑永远优先写库，不等待网络返回
- 同步采用“增量推送 + 增量拉取 + 周期性快照校准”
- 删除采用软删除，保证跨端传播和冲突追踪
- 所有时间统一使用 UTC
- 版本号基于对象维度递增，不共用全局版本号

### 5.2 协议域

在现有远程控制协议中新增 notes 域，不混用聊天实体。

建议请求：
- `notes_snapshot`
- `notes_pull_since`
- `notes_push_ops`
- `notes_subscribe`
- `notes_ack`
- `notes_ping`

建议事件：
- `notes_changed`
- `notes_conflict`
- `notes_sync_status`

### 5.3 同步流程

#### 启动或进入笔记页

1. 打开本地 SQLite
2. 加载本地数据并渲染
3. 读取 `last_sync_cursor`
4. 在线时请求 `notes_pull_since`
5. 若游标失效或数据校验失败，则回退到 `notes_snapshot`

#### 本地写入

1. 用户执行新增、编辑、删除、重命名、置顶、置底、导入等动作
2. 本地数据库事务提交
3. 生成 outbox 记录
4. 更新界面
5. 在线时立即尝试推送

#### 在线推送

1. 从 outbox 取 `pending` 记录
2. 发送 `notes_push_ops`
3. 电脑端应用成功后返回 ack
4. 手机端将对应 outbox 标记为 `acked`
5. 电脑端再广播 `notes_changed` 给已连接端

#### 周期性校准

为避免事件丢失，建议在以下时机触发轻量校准：
- 进入笔记页
- 应用从后台恢复前台
- WebSocket 重连成功后
- 每隔固定时间，例如 60 秒

## 6. 冲突处理

### 6.1 冲突判定

在以下情况下判定为冲突：
- 同一 `Notebook` 或 `NoteEntry`
- 双端基于相同 `base_version` 分别修改
- 服务端合并时发现关键字段都已变化

关键字段包括：
- notebook：`title`、`pinned`、`sort_order`
- entry：`content`、`pinned`、`sort_order`

### 6.2 冲突策略

冲突时不弹窗阻断，不做最后写入覆盖，统一生成冲突副本。

#### notebook 冲突

- 保留当前原始 notebook
- 新建一个 notebook 副本
- 标题后缀增加 `（冲突副本-手机）` 或 `（冲突副本-电脑）`
- `is_conflict_copy = 1`
- `origin_notebook_id` 指向原 notebook

#### entry 冲突

- 保留当前原始 entry
- 在同一 notebook 下新建一条 entry 副本
- 内容前插入一行来源说明，例如 `【冲突副本：来自手机端】`
- `is_conflict_copy = 1`
- `origin_entry_id` 指向原 entry

### 6.3 正在编辑时收到远程更新

如果某条 entry 正在编辑，期间收到另一端更新：
- 不直接覆盖编辑框内容
- 在界面中显示轻提示
- 当前用户继续保存时，如果基线版本已变化，则生成冲突副本

## 7. 电脑端设计

### 7.1 UI 结构

电脑端当前以 `wxPython` 主窗口、列表控件、编辑框和全局键盘事件为主。笔记功能应复用这一模式，不单独引入复杂控件体系。

建议新增根视图状态：
- `chat`
- `notes`

笔记子状态：
- `notes_list`
- `note_detail`
- `note_edit`

建议布局：
- 主界面保留现有聊天区
- 增加一个笔记视图入口，切换到 notes 根视图后显示笔记相关内容
- 笔记列表显示 notebook 项目
- 打开 notebook 后显示该 notebook 的 entry 列表
- 打开 entry 后显示多行编辑框

### 7.2 笔记列表交互

要求：
- 主界面上增加笔记列表
- 列表中每个 notebook 对应一个列表项目
- 上下方向键在笔记之间切换
- Enter 打开选中 notebook 的详情界面
- Application 菜单键打开菜单

菜单项：
- 添加笔记
- 删除笔记
- 搜索笔记
- 重命名笔记

### 7.3 笔记详情交互

要求：
- 笔记详情界面仍使用列表形式
- 每条 `NoteEntry` 对应一个列表项目
- 上下方向键在条目间切换
- Enter 打开条目编辑界面
- Application 菜单键打开菜单

菜单项：
- 添加笔记
- 删除笔记
- 编辑笔记
- 置顶笔记
- 置底笔记
- 从文件导入
- 从剪贴板导入

### 7.4 笔记编辑交互

要求：
- 编辑界面为多行编辑框
- `Ctrl+Enter` 或 `Alt+S` 保存并退出到笔记详情界面
- `Esc` 退出编辑界面
- 若内容已变化，退出时弹对话框：
  - 保存
  - 不保存
  - 取消

### 7.5 焦点与 Tab

要求：
- 无论当前位于笔记列表、笔记详情或笔记编辑界面，按 Tab 都必须能切换到主界面其他控件
- 不应通过强拦截 Tab 打破 wx 默认焦点链
- 需要为笔记相关控件补全 tab order，使其与输入框、回答列表、历史聊天列表保持可遍历

### 7.6 状态恢复

电脑端下次启动时要恢复笔记界面状态。

需恢复内容：
- 当前根视图是否为 `notes`
- 当前笔记子状态是列表、详情还是编辑
- 当前 notebook ID
- 当前 entry ID
- 未保存草稿
- 编辑框光标位置
- 编辑框滚动位置

恢复规则：
- 若 entry 仍存在，恢复到原编辑界面
- 若 entry 已不存在但 notebook 仍存在，退回详情界面
- 若 notebook 也不存在，退回笔记列表界面

### 7.7 导入功能

导入入口位于笔记详情菜单中。

#### 从文件导入

- 弹出文件选择对话框
- 读取文本文件
- 编码优先 UTF-8，失败后可尝试常见本地编码
- 逐行切分
- 每个非空行生成一个新的 `NoteEntry`
- `source = import_file`

#### 从剪贴板导入

- 读取剪贴板纯文本
- 逐行切分
- 每个非空行生成一个新的 `NoteEntry`
- `source = import_clipboard`

导入操作作为单次事务写库并批量加入 outbox。

### 8.1 根导航调整

底部标签结构：
- 聊天

### 8.2 笔记列表页

要求：
- 左上角有“新建笔记”按钮
- 点击笔记进入详情页
- 长按笔记弹出菜单：
  - 删除

多选模式要求：
- 用户选择多条 notebook
- 页面底部提供：
  - 全选
  - 删除

### 8.3 笔记详情页

要求：
- 每条 `NoteEntry` 按列表排列
- 点击条目进入编辑页
- 长按条目弹出删除确认框

页面底部提供两个入口：
- 文字输入按钮
- 语音输入按钮

### 8.4 条目编辑页

要求：
- 使用多行编辑框
- 底部提供：
  - 保存
  - 复制

行为：
- 点击保存，保存并退出到详情页
- 点击复制，将内容写入剪贴板并退出

### 8.5 文字输入

- 点击文字输入按钮，弹出输入框或进入录入界面
- 输入完成后保存为新的 `NoteEntry`
- 默认 `source = manual`

### 8.6 语音输入

- 长按语音按钮开始录音
- 松开结束录音并获取识别结果
- 识别结果使用豆包 `doubao-seed-2-0-mini-260215` 做轻量润色
- 润色后生成新的 `NoteEntry`

润色原则：
- 去除明显口头禅
- 补充基本标点
- 提升通顺度
- 不改变事实含义
- 模型失败时回退为原始识别文本

`source = voice`

## 11. 错误处理与边界规则

需明确以下规则：

- 删除使用软删除，不做硬删除
- 搜索先做本地搜索，不做跨端远程搜索
- 置顶和置底本质是更新 `sort_order`
- 同步失败不影响本地保存
- outbox 支持断网重试
- 同步游标失效时允许回退到全量快照
- 已删除 notebook 下的 entry 不再单独恢复显示
- 恢复界面时若对象不存在，按层级向上回退
- 手机端多选删除先只支持 notebook 层，不扩展到 entry 层

## 14. 决策摘要

本设计确定以下关键决策：
- 两端均使用 SQLite 作为笔记主存储
- 电脑端继续作为同步中枢
- 同步采用增量事件 + 快照校准
- 两端都支持离线编辑
- 冲突采用“保留两份，生成冲突副本”
- UI 状态与业务数据分离保存
- 先本地功能，后同步，最后冲突处理

该设计与现有 `wxPython` 桌面项目和 Flutter 手机项目兼容，能在不引入额外服务部署的前提下实现双端笔记与跨端同步。
