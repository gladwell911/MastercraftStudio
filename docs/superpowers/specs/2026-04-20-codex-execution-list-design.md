# Codex 执行过程列表设计

日期：2026-04-20

## 背景

当前聊天界面右侧只展示问答内容。Codex CLI 在执行过程中会持续产生中间事件，但程序只在任务完成且最终回答生成后才把结果呈现给用户，导致用户无法实时了解当前执行到了哪一步。

本次改动的目标是在单个聊天内提供一个可切换的“执行过程列表”视图，让用户可以在不打断当前任务的前提下查看 Codex CLI 的实时执行步骤，并在任务结束后保留这些步骤记录。

## 用户目标

- 在某个聊天中按 `F1`，在“回答列表”和“执行过程列表”之间切换。
- 执行过程按列表项逐条展示，每出现一个新步骤就在列表尾部追加一个新项目。
- 新步骤出现时不抢占键盘焦点，不改变用户当前焦点所在控件。
- 执行过程需要随聊天持久化保存，重新打开聊天后仍可查看。

## 非目标

- 不改变左侧历史聊天列表的结构和行为。
- 不将执行过程混入现有回答列表。
- 不把 `agent_message_delta` 这样的字符级流式输出逐条写入执行列表。
- 不新增弹窗式过程监控界面。

## 现状

- 右侧问答区当前由 `answer_list` 承载，使用 `wx.ListBox` 渲染。
- Codex 事件通过 `CodexAppServerClient` 持续上报到 UI 层，事件类型包括 `item_started`、`item_completed`、`plan_updated`、`diff_updated`、`stderr`、`server_request`、`turn_started`、`turn_completed` 等。
- 现有逻辑主要把最终回答写入回答列表，没有单独维护执行步骤列表。

## 方案概览

采用“双列表控件，同区域切换显示”的方案：

- 保留现有 `answer_list`。
- 新增一个与其并列、位置相同的 `execution_list`。
- 右侧列表区域同一时刻只显示一个列表。
- 每个聊天维护自己的详细视图模式：`answers` 或 `execution`。
- Codex 中间事件映射为可读的执行步骤，追加到当前聊天的 `execution_steps` 中。

这样可以保持回答列表逻辑稳定，同时把执行过程视图独立出来，避免把两类数据混在一起。

## UI 设计

### 右侧列表区

右侧现有“回答：”区域扩展为一个可切换详情区：

- 当模式为 `answers` 时，标题显示“回答：”，展示 `answer_list`。
- 当模式为 `execution` 时，标题显示“执行过程：”，展示 `execution_list`。

两个列表控件共享同一块布局区域，通过显示/隐藏切换，不同时展示。

### 键盘行为

- 在聊天界面按 `F1` 时切换当前聊天的详情视图模式。
- 切换只影响右侧列表区域，不影响输入框内容、发送状态、左侧聊天选中状态。
- 列表追加新步骤时不主动调用 `SetFocus()`，不改变当前焦点。

### 视图记忆

每个聊天记录自己的详情视图模式：

- 用户在聊天 A 切到执行过程列表后，离开再回来，仍保持执行过程列表。
- 老数据默认使用 `answers`。

## 数据模型

### 聊天级字段

在每个聊天状态中新增以下字段：

- `detail_panel_mode: str`
  - 取值：`answers` 或 `execution`
  - 默认：`answers`
- `execution_steps: list[dict]`
  - 默认：空列表

### 执行步骤结构

每个步骤保存以下字段：

- `type`: 事件类别，如 `turn_started`、`item_started`、`item_completed`、`plan_updated`
- `text`: 列表展示文本
- `phase`: 可选，来自事件 `phase`
- `status`: 可选，来自事件 `status`
- `thread_id`: 可选
- `turn_id`: 可选
- `item_id`: 可选
- `created_at`: 时间戳

该结构以“可显示、可持久化、可追溯”为目标，不追求完整保存原始协议包。

## 事件映射规则

执行列表是“步骤流”，不是“字符流”。因此只接收有明确边界的事件。

### 记录为步骤的事件

- `turn_started`
  - 追加：`开始处理本轮请求`
- `item_started`
  - 追加：根据事件内容生成可读文本，例如 `开始执行：<status或phase>`
- `item_completed`
  - 若 `phase == final_answer`，追加：`已生成最终回答`
  - 其他情况追加：`完成执行：<status或phase>`
- `plan_updated`
  - 追加：`计划更新：<explanation>`
- `diff_updated`
  - 追加：`已生成代码变更`
- `stderr`
  - 追加 stderr 文本，必要时可加前缀 `错误输出：`
- `server_request`
  - 追加：`等待用户输入`
  - 如果后续需要区分选择/审批，可继续细分展示文本
- `turn_completed`
  - 追加：`本轮处理结束`

### 不记录为步骤的事件

- `agent_message_delta`
  - 不进入执行列表，避免产生碎片化字符流项目
- 与当前需求无关的低价值通知事件

## 追加与刷新策略

- 每当收到新的可记录事件，向目标聊天的 `execution_steps` 末尾追加一条步骤。
- 如果该聊天当前正在显示执行过程列表，则同步刷新 `execution_list`。
- 如果该聊天不在前台或当前显示的是回答列表，仍然保存步骤，但不抢焦点。
- 后台聊天收到步骤事件时，只更新其持久化状态；用户切回该聊天后再完整渲染。

## 焦点与滚动行为

- 新步骤追加时不改变当前焦点。
- 如果执行列表拥有焦点或当前聊天正显示执行列表，可将列表选择保持在原位，不因为新增步骤强制改选。
- 若未来需要“自动滚动到末尾”，应作为独立增强，不纳入本次范围。

## 持久化与兼容性

### 状态保存

在现有聊天状态序列化中增加：

- `detail_panel_mode`
- `execution_steps`

### 向后兼容

加载旧状态时：

- 缺少 `detail_panel_mode` 时，默认值为 `answers`
- 缺少 `execution_steps` 时，默认值为 `[]`

这样不会影响旧聊天和现有持久化文件读取。

## 代码改动范围

预计涉及以下区域：

- `main.py`
  - UI 构建：新增 `execution_list` 与详情标题切换
  - 状态加载/保存：增加新字段
  - 键盘事件：处理 `F1` 切换
  - 渲染逻辑：新增执行列表渲染函数
  - Codex 事件处理：把中间事件映射为执行步骤并写入聊天状态
- `tests/test_main_unit.py`
  - 增加视图切换、状态持久化、执行步骤渲染、焦点稳定性等测试

## 错误处理

- 若收到空文本或无法生成可读标题的事件，使用保底文案，例如 `执行步骤更新`
- 若事件来自已不存在的聊天，忽略 UI 渲染，但不应导致主流程异常
- 若执行列表刷新失败，不影响回答列表主流程与任务执行

## 测试策略

需要覆盖以下场景：

- 默认显示回答列表，按 `F1` 切到执行过程列表，再按一次切回
- `F1` 切换后详情标题随模式变化
- 新执行步骤追加到列表末尾
- 追加步骤时不触发焦点跳转
- 执行步骤会写入聊天状态并随保存/加载恢复
- 后台聊天收到执行步骤后，切回时能看到完整列表
- `agent_message_delta` 不进入执行列表
- `final_answer` 仍按现有逻辑进入回答列表，同时执行列表追加“最终回答已生成”步骤

## 取舍

本设计不直接复用 `answer_list` 来承载两种数据，而是分离出 `execution_list`。这样会增加一套列表控件和元数据，但可以显著降低对现有回答渲染逻辑的侵入，后续如果要给执行步骤增加详情查看、复制原始事件文本等功能，也更容易扩展。

## 成功标准

以下条件同时满足则视为完成：

- 用户可在聊天内按 `F1` 在回答列表与执行过程列表之间切换
- Codex 执行中产生的新步骤能实时追加到执行过程列表末尾
- 新步骤追加不会抢夺焦点
- 执行过程会持久化到聊天记录，重新打开后可查看
- 现有回答列表与最终回答展示行为不回归
